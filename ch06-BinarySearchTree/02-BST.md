## 2. 이진 탐색 트리 (Binary Search Tree, BST)
이진 탐색 트리는 **이진 탐색(Binary Search)** 의 효율적인 탐색 로직을 트리 구조에 접목하여,   
데이터의 삽입과 삭제가 빈번한 상황에서도 빠른 검색을 유지하도록 설계된 자료구조입니다.  

### 2-1. 정의 및 성질 (Formal Definition)  
이진 탐색 트리는 이진 트리(Binary Tree)의 일종으로, 모든 노드 $n$에 대해 다음의 **수학적 조건** 을 만족해야 합니다.

* **좌측 서브트리 조건**: 노드 $n$의 왼쪽 서브트리에 있는 모든 노드의 키값은 노드 $n$의 키값보다 **작아야 함** ( $Key(L) < Key(n)$ )  
* **우측 서브트리 조건**: 노드 $n$의 오른쪽 서브트리에 있는 모든 노드의 키값은 노드 $n$의 키값보다 **커야 함** ( $Key(R) > Key(n)$ )  
* **재귀적 성질**: 모든 서브트리 역시 그 자체로 이진 탐색 트리여야 함  

### 2-2. 주요 특징과 탐색 기법  
이진 탐색 트리는 데이터가 정렬된 상태로 유지되는 특성 덕분에 다음과 같은 유용한 성질을 가집니다.  

#### ① 중위 순회(Inorder Traversal)와 정렬  
이진 탐색 트리를 **중위 순회(Inorder Traversal, L->V->R)** 하면 저장된 키값들이 **오름차순으로 정렬**되어 출력됩니다.  
* 이는 별도의 정렬 알고리즘 없이도 정렬된 데이터를 얻을 수 있음을 의미합니다.  

#### ② 최소값과 최대값 찾기  
* **최소값(Minimum)**: 루트에서 시작하여 `null`을 만날 때까지 계속해서 **왼쪽 자식**을 따라 내려가면 만나는 노드입니다.  
* **최대값(Maximum)**: 루트에서 시작하여 `null`을 만날 때까지 계속해서 **오른쪽 자식**을 따라 내려가면 만나는 노드입니다.  

### 2-3. 트리 구조에 따른 효율성 분석  
이진 탐색 트리의 성능은 트리의 **높이($H$)** 에 직접적으로 의존합니다. 동일한 데이터 세트라도 삽입되는 순서에 따라 트리의 모양과 성능이 극명하게 갈립니다.  

| 트리 모양 | 특징 | 탐색 시간 복잡도 |  
| :--- | :--- | :--- |  
| **최선의 모양 (Balanced)** | 좌우 균형이 완벽하게 잡힌 상태 | **$O(\log N)$** |  
| **일반적인 모양** | 어느 정도 균형이 잡힌 상태 | $O(H)$ |  
| **최악의 모양 (Skewed)** | 한쪽으로 치우친 선형 구조 (연결 리스트와 동일) | **$O(N)$** |  

#### 시간 복잡도와 높이의 관계  
* 트리가 균형을 이룰 때, 노드 수가 $N$이면 높이 $H \approx \log_2 N$이 됩니다.  
* 하지만 정렬된 데이터를 그대로 삽입할 경우 트리가 한쪽으로 길게 늘어지는 **경사 이진 트리(Skewed Tree)** 가 형성되어, 검색 효율이 배열의 순차 탐색 수준으로 떨어지게 됩니다.  

### 2-4. 이진 탐색(Array) vs 이진 탐색 트리(Tree)  
둘 다 이진 탐색 원리를 사용하지만, 데이터 유지 관리 측면에서 차이가 있습니다.  

* **이진 탐색 (정렬 배열)**: 고정된 크기 내에서 매우 빠르지만($O(\log N)$), 새로운 데이터를 삽입하거나 삭제할 때 배열의 요소를 밀거나 당기는 $O(N)$의 비용이 발생합니다.    
* **이진 탐색 트리**: 트리가 균형을 유지한다는 전제하에 **탐색, 삽입, 삭제 모두 $O(\log N)$**에 수행할 수 있어 동적인 데이터 관리에 훨씬 유리합니다.  

### 3-1. 이진 탐색 트리의 클래스 구조 (Implementation)  
Java를 기준으로 한 이진 탐색 트리의 기본적인 설계입니다.  

#### ① 노드(Node) 클래스  
트리의 각 마디를 구성하며, 데이터와 연결 정보를 담습니다.  
* **id**: 탐색의 기준이 되는 **키(Key)** 값입니다. (비교를 위해 `Comparable` 인터페이스 구현 필요)  
* **name**: 키와 연관된 실제 **값(Value)** 데이터입니다.  
* **left / right**: 왼쪽 및 오른쪽 자식 노드를 가리키는 레퍼런스(포인터)입니다.  

#### ② BST 클래스  
전체 트리를 관리하는 클래스로, 트리의 시작점인 **루트(root)** 노드를 멤버 변수로 가집니다.  

## 3. 이진 탐색 트리의 구현
이진 탐색 트리를 실제 프로그램으로 옮길 때 필요한 클래스 구조와 설계 원칙을 상세히 정리합니다.  

### 3-1. 노드(Node) 클래스의 세부 설계  
트리의 가장 기초 단위인 노드는 데이터와 포인터를 포함하는 복합 구조체로 설계됩니다.  

<img width="418" height="183" alt="image" src="https://github.com/user-attachments/assets/2cfe5d04-6ee2-4417-bcb0-ef9253e5f9ea" />

* **키(id)**: 데이터를 식별하고 비교하는 기준입니다. 이진 탐색 트리의 정렬 기준이 됩니다.  
* **값(name)**: 해당 키와 연결된 실제 데이터(정보)입니다.   
* **좌측 자식 포인터(left)**: 자신보다 작은 키값을 가진 자식 노드를 가리킵니다.   
* **우측 자식 포인터(right)**: 자신보다 큰 키값을 가진 자식 노드를 가리킵니다. 

| 필드명 | 역할 | 특징 | 
| :--- | :--- | :--- | 
| **Key id** | 탐색 키 | `Comparable` 인터페이스를 상속받아 비교 가능해야 함 | 
| **Value name** | 실제 데이터 | 제네릭 타입을 통해 다양한 객체 저장 가능 | 
| **Node left** | 왼쪽 연결 | 현재 노드의 키보다 작은 서브트리의 루트를 지칭 | 
| **Node right** | 오른쪽 연결 | 현재 노드의 키보다 큰 서브트리의 루트를 지칭 | 

### 3-2. 제네릭(Generic)과 비교 인터페이스 활용  
이진 탐색 트리는 다양한 타입의 데이터를 처리할 수 있도록 제네릭을 사용하여 설계하며, 키값의 선후 관계를 판별하기 위해 인터페이스를 활용합니다.  

1.  **Key extends Comparable<Key>**:   
    * 트리 구조를 형성할 때 "어느 쪽 자식으로 보낼 것인가"를 결정하기 위해 반드시 `compareTo()` 메소드를 사용할 수 있는 타입이어야 합니다.  
2.  **동적 바인딩**:  
    * `Comparable`을 통해 정수, 실수뿐만 아니라 사용자 정의 객체(예: 사번, 학생번호)도 키로 사용할 수 있습니다.  

### 3-3. 트리(BST) 관리 클래스의 역할  
전체적인 트리의 상태를 유지하고 연산을 수행하는 핵심 컨트롤러 역할을 합니다.  

* **Root 노드 관리**: 트리의 시작점인 `root` 포인터를 유일하게 관리합니다. 모든 연산(탐색, 삽입, 삭제)은 이 `root`에서 시작됩니다.  
* **연산 메소드 캡슐화**: 외부에서는 `get()`, `put()` 같은 단순한 인터페이스만 호출하고, 실제 재귀적인 처리는 클래스 내부의 `private` 메소드에서 처리하도록 설계합니다.  
* **트리 상태 모니터링**: 트리가 비어있는지(`root == null`), 특정 조건을 만족하는지 확인하는 기능을 포함합니다.  

### 3-4. 구현 시 핵심 설계 포인트  
효율적이고 안전한 자료구조 구현을 위해 다음 사항들을 준수합니다.  

1.  **비교 로직의 표준화**:   
    * 산술 연산자(`<`, `>`) 대신 반드시 **`compareTo()`**의 결과값(음수, 0, 양수)을 기준으로 흐름을 제어합니다.  
2.  **재귀 vs 반복문 선택**:  
    * 트리의 높이가 깊지 않을 때는 **재귀(Recursion)** 방식이 코드가 간결하고 논리적으로 명확합니다.  
    * 매우 큰 데이터를 다룰 때는 스택 오버플로우를 방지하기 위해 **반복문(Iterative)** 방식을 고려할 수도 있습니다.  
3.  **캡슐화(Encapsulation)**:   
    * 노드의 내부 필드(`left`, `right`)는 `private`으로 선언하여 외부의 직접 수정을 막고, 접근자(Getter/Setter)를 통해 데이터 무결성을 유지합니다.  
4.  **Null 처리**:  
    * 자식이 없는 단말 노드의 `left`, `right` 필드는 항상 `null`을 가리키도록 설정하여 트리의 끝을 명확히 표시합니다.
