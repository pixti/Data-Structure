## 5. 상속(Inheritance) vs 포함(Composition)

객체지향 프로그래밍(OOP)에서 클래스 간의 관계를 설정하는 두 가지 핵심 방법입니다.

---
## 5-1. 상속 (Inheritance) - "IS-A" 관계
부모 클래스의 성질을 자식이 물려받아 기능을 확장( `extends` )하는 수직적 관계입니다.

* **핵심 철학**: "자식은 부모의 한 종류이다 (**IS-A**)"
* **예시**:
    - 학생은 사람이다. (Student **is a** Person)
    - 연결 리스트 스택은 연결 리스트의 일종이다. (LinkedStack **is a** LinkedList)
* **장점**:
    - **코드 재사용**: 부모가 가진 기능을 자식이 그대로 사용하므로 중복 코드가 줄어듭니다.
    - **계층 구조**: 세상을 체계적으로 분류하여 관리할 수 있습니다.

## 5-2. 포함/구성 (Composition) - "HAS-A" 관계
한 클래스가 다른 클래스를 자신의 멤버 변수(부품)로 가지고 있는 수평적 조립 관계입니다.

* **핵심 철학**: "전체는 부분(부품)을 가지고 있다 ( **HAS-A** )"
* **예시**:
    - 자동차는 엔진을 가지고 있다. (Car **has an** Engine)
    - 대학교는 컴퓨터공학과를 가지고 있다. (University **has a** CS Department)
* **장점**:
    - **유연함**: 상속은 한 번 맺으면 끊기 힘든 '혈연관계'라면,  
      포함은 언제든 부품을 갈아 끼울 수 있는 '조립관계'입니다. (예: 학과 추가/삭제 시 멤버 변수만 수정하면 끝)

---
# 6. 상속(Inheritance)을 이용한 LinkedList Stack 구현

`LinkedList` 를 부모로 삼아 스택을 만드는 방식입니다.

1. **코드 재사용 (Code Reuse)**: `LinkedList`가 이미 가진 강력한 기능(`addFirst`, `removeFirst`, `isEmpty` 등)을 그대로 물려받아 스택의 핵심 로직을 직접 짤 필요가 없습니다.
2. **`super` 키워드의 역할**: `super.addFirst(newItem)` 처럼 사용하면,  이 메서드가 자식이 아닌 **부모 클래스에서 정의된 것**임을 명확히 합니다.  
  메서드 재정의(Overriding) 시 부모의 기능을 강제로 호출할 때도 필수적입니다.
3. **구조적 차이**:
    - **이전 방식 (Composition)**: 리스트를 부품으로 '소유'하는 방식.  
    - **상속 방식 (Inheritance)**: 클래스 자체가 리스트가 되는 방식.  
* **주의점**: 상속을 받으면 스택에 어울리지 않는 부모의 메서드(중간 삽입/삭제 등)까지 외부로 노출되어 **캡슐화**가 깨질 위험이 있습니다.

# 7. 직접 구현 vs 상속 구현: 메서드 탐색과 설계의 무결성

## 7-1. 메서드 탐색 우선순위 (Method Resolution)
자바는 메서드 호출 시 **"나부터 찾고, 없으면 부모에게 간다"** 는 원칙을 따릅니다.
1. **1순위 (나 자신)**: `InheritedStack` 에 메서드가 있는지 확인.
2. **2순위 (부모 클래스)**: 없다면 `extends` 한 `LinkedList` 로 올라가서 탐색.

## 7-2. 인덱스 기반 접근(add, get)의 위험성
기술적으로 부모의 `get(3)` 메서드를 쓰는 것은 가능하지만, **설계적으로는 치명적인 실수** 입니다.

1. **ADT의 무결성 파괴**: 스택의 핵심 규칙은 **LIFO(오직 Top만 본다)** 입니다. 인덱스 접근을 허용하는 순간 스택의 정체성이 무너집니다.
2. **캡슐화 위반**: 내부 구현을 숨겨야 하는데, 사용자가 중간 데이터를 건드릴 수 있게 되면 예기치 못한 버그의 원인이 됩니다.

> **[자판기 비유]**
> 자판기(스택)는 돈을 넣고 음료를 뽑는 입구만 있어야 합니다.  
> 그런데 상속을 잘못 받아 옆면(인덱스 접근)이 뚫려있다면? 누구나 옆을 열어 음료를 가져가게 되고,  
> 자판기라는 시스템(ADT)은 유지될 수 없습니다.

# 8. 포함(Composition)을 이용한 LinkedList Stack 구현
현대 프로그래밍에서 상속보다 권장되는 **"HAS-A"** 기반의 구현 방식입니다.

## 8-1. 포함 관계와 위임 (Delegation)
스택이 리스트가 되는 것이 아니라, **리스트를 부품으로 내부에 숨겨두는 구조**입니다.

* **위임(Delegation)**: 사용자가 `push()`를 호출하면,
  스택이 직접 일하지 않고 내부의 리스트 부품에게  
  " **네가 가진 `addFirst` 기능 좀 써서 이거 좀 넣어줘** " 라고 일을 시키는 행위입니다.
    - `push(item)` → `list.addFirst(item)` 호출
    - `pop()` → `list.removeFirst()` 호출 후 결과 반환

## 8-2. 왜 포함(Composition)이 상속보다 우월한가?

1. **완벽한 정보 은닉 (Encapsulation)**:
   - 내부의 `LinkedList`를 `private`으로 선언하여 외부에서는 스택 안에 무엇이 들어있는지 알 수 없게 차단합니다. 오직 스택의 인터페이스(`push`, `pop`, `top`)만 보여줍니다.
2. **낮은 결합도 (Low Coupling)**:
   - 내부 부품을 `LinkedList`에서 `ArrayList`로 바꾸더라도 외부에서 사용하는 메서드는 바뀔 필요가 없습니다. 부품만 싹 갈아 끼우면 되는 유연한 구조입니다.

# 9. 자바에서의 프로그램 수행과 메모리 구조 : 가상 메모리(Virtual Memory)

프로그램이 실행될 때 가상 메모리(Virtual Memory)는  
크게 **정적 영역(Static)** 과 **동적 영역(Dynamic)** 으로 나뉘어 관리됩니다.
자료구조로 배운 '스택' 원리가 운영체제 레벨에서 어떻게 쓰이는지 정리합니다.

## 9-1. 메모리의 4층 구조

| 분류 | 영역 | 설명 |
| :--- | :--- | :--- |
| **Dynamic (동적 영역)** | **Heap** | 수행 중에 사용자가 할당받는 메모리. (예: Java의 `new`, C의 `malloc()`, 파이썬의 객체 생성) |
| **Dynamic (동적 영역)** | **Stack** | **Active 상태**의 함수 관련 파라미터(매개변수)와 지역 변수들이 저장되는 공간. |
| **Static (정적 영역)** | **Data** | 프로그램이 끝날 때까지 유지되는 데이터. (예: 전역 변수, 정적(Static) 변수) |
| **Static (정적 영역)** | **Code** | 프로그램의 실제 수행 코드가 기계어로 저장되는 영역. |

## 9-2. 실행 시간 스택 (Runtime Stack)

운영체제는 함수 호출을 관리하기 위해 **LIFO(후입선출)** 원리를 사용하는 'Stack' 영역을 운용합니다.

* **함수 호출 (Push)**: 새로운 함수가 호출되면 해당 함수의 변수와 복귀 주소를 담은 '스택 프레임'이 스택 위에 쌓입니다.
* **함수 종료 (Pop)**: 함수 수행이 끝나면 가장 위에 있던 해당 스택 프레임이 제거되면서 호출했던 지점으로 돌아갑니다.
* **특징**: 가장 나중에 호출된 함수가 가장 먼저 끝나야 하는 프로그램의 생리상 스택 구조가 가장 적합합니다.

## 9-3. Heap과 Stack의 동적 상호작용
Heap과 Stack은 하나의 제한된 메모리 공간을 나누어 사용하며 메모리의 서로 반대 방향으로 자라납니다.

1. **공간 공유**: 
   - **Stack**: 높은 주소에서 낮은 주소 방향으로 쌓입니다.
   - **Heap**: 낮은 주소에서 높은 주소 방향으로 할당됩니다.
2. **충돌과 에러**:
   - **Stack Overflow**: 함수 호출이 너무 깊어져 스택이 힙 영역을 침범할 때 발생합니다.
   - **OutOfMemory (Heap)**: 객체를 너무 많이 생성하여 힙이 스택 영역을 침범할 때 발생합니다.
3. **참조 관계**:
   - 우리가 선언한 **참조 변수(이름표)** 는 **Stack**에 저장됩니다.
   - `new`로 생성한 **실제 객체(데이터)** 는 **Heap**에 저장됩니다.
   - Stack에 있는 변수가 Heap에 있는 객체의 주소를 가리키는(Pointing) 구조입니다.

> * **Stack**: "지금 당장 하는 일(함수)"을 처리하기 위한 임시 작업대.
> * **Heap**: "오랫동안 들고 있을 물건(객체)"을 보관하는 자유로운 창고.
> * 이 두 영역은 동적으로 크기가 변하며, 서로의 영역을 침범하지 않도록 운영체제가 관리한다.
