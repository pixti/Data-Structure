## 1-1. 행렬(Dense Matrix) - 2차원 배열으로 표현
수학의 행렬(Matrix)을 컴퓨터 메모리에 담는 가장 기본적이고 직관적인 방법입니다.

* **구조**: `a[m][n]` 형태의 2차원 배열을 할당합니다.
* **물리적 저장**: 메모리는 1차원이므로, 행렬은 행 우선 표기법(Row-major order)에 따라 `0행 -> 1행 -> 2행...` 순서로 일렬로 늘어서 저장됩니다.
* **특징**: 모든 칸(0 포함)에 메모리 공간을 할당합니다.

**장점과 단점**  
 **장점**: 인덱스 `(i, j)`만 알면 주소 계산 공식 ($a + (i \times \text{열의 개수} + j) \times \text{크기}$) 을 통해 **$O(1)$** 만에 즉시 접근(Random Access)이 가능합니다.  
 
 **단점 (공간)**: 데이터가 거의 없는 행렬도 모든 칸을 차지하므로 메모리 낭비가 심합니다.  
 $1000 \times 1000$ 행렬에 데이터가 10개뿐이어도 100만 개의 메모리 공간이 필요.
 
 **단점 (시간)**: 데이터의 유무와 상관없이 항상 $O(N^3)$의 시간이 걸리므로, 행렬이 커질수록 연산량이 폭발적으로 증가합니다.  
 행렬 곱셈 시 $1000^3$ 번(10억 번)의 연산을 수행해야 하는데, 대부분이 0 * 0을 계산하는 무의미한 과정  

### 행렬 곱셈이 $O(N^3)$인 이유
```java
for (int i = 0; i < mat1.rows; i++) {
    for (int j = 0; j < mat2.cols; j++) {
        sum = 0;
        for (int k = 0; k < mat1.cols; k++) {
            sum += (mat1[i][k] * mat2[k][j]);
        }
        result[i][j] = sum;
    }
}
```

일반적인 $N \times N$ 정방행렬 두 개를 곱할 때, 시간 복잡도가 $O(N^3)$이 되는 원인은 다음과 같습니다.

1.  **결과 행렬의 크기 ($N^2$)**: $N \times N$ 행렬끼리 곱하면 결과물도 $N \times N$ 행렬이 나옵니다. 즉, 우리가 채워야 할 결괏값의 칸은 총 **$N^2$개**입니다.
2.  **한 칸을 채우는 연산 ($N$)**: 결과 행렬의 특정 위치 $(i, j)$에 들어갈 값을 구하려면, 첫 번째 행렬의 $i$번째 행(원소 $N$개)과 두 번째 행렬의 $j$번째 열(원소 $N$개)을 하나씩 짝지어 곱한 뒤 모두 더해야 합니다. 이 과정에서 **$N$번의 곱셈**이 필수적으로 일어납니다.
3.  **최종 연산량**: (채워야 할 칸의 수 $N^2$) $\times$ (한 칸당 필요한 연산 $N$) = **총 $N^3$번의 연산**이 수행됩니다.

## 2-1. 효율적인 희소 행렬(Sparse Matrix) 표현
희소 행렬(sparse matrix)은 0이 아닌 원소가 극히 일부이고, 0이 대다수인 행렬을 의미합니다.
대부분의 항이 0인 행렬에서 0이 아닌 유효한 항(Non-zero term)만 골라 관리하는 방식으로  
기존처럼 index에 위치를 숨기지 않고, 행(row)/열(col) 번호를 직접 명시한다면 시간/공간이 절약 됩니다.

## 2-2. 데이터 구조 구현 방식(Implementation Detail) - 3원소 쌍(Triple) 방식

* **MatrixTerm 클래스**: 개별 항의 데이터를 담는 최소 단위입니다.  
* <행(row), 열(col), 값(value)> 3원소 쌍으로 유일하게 식별합니다.
 ```java
public class MatrixTerm {
    private int row;    // 행 번호
    private int col;    // 열 번호
    private int value;  // 실제 값
}
```  

* **SparseMatrix 클래스**: MatrixTerm들 배열에 담아 전체 행렬을 관리하는 틀입니다.
  - `rowSize`, `colSize`: 전체 행렬의 규격
  - `termCnt`: 유효한 데이터의 총 개수
  - `array`: MatrixTerm 객체들을 담는 배열

```java

public class SparseMatrix {
    private int rowSize;        // 전체 행렬의 행 수
    private int colSize;        // 전체 행렬의 열 수
    private int termCnt;        // 0이 아닌 항의 총 개수
    private MatrixTerm[] array; // MatrixTerm 객체들을 담는 3원소 쌍(Triple) 배열
}
```

## 2-3. 저장 원칙: "엄격한 오름차순 약속"
희소 행렬이 효율적인 연산 성능을 내기 위해서는 데이터를 저장할 때 다음 규칙을 반드시 지켜야 합니다. 

1. 제1원칙 (행 오름차순): 행(row) 인덱스가 낮은 것에서 높은 순서로 저장합니다.  
2. 제2원칙 (열 오름차순): 행 번호가 같다면, 열(col) 인덱스가 낮은 것에서 높은 순서로 저장합니다.
       
**이유** : 이 약속이 지켜져야만 덧셈 연산 시 두 행렬을 앞에서부터 한 번만 훑는 $O(\text{termCnt})$ 의 선형 시간 연산이 가능해지기 때문입니다.

## 3-1. 희소 행렬의 덧셈 (Add)
0이 아닌 항들만 추려내어 더하기 때문에 매우 효율적입니다.

* **로직**: 다항식의 덧셈(Two Pointers)과 유사하게 두 행렬의 항들을 순차적으로 비교하며 나아갑니다.
* **수수행 시간**: 두 행렬의 0이 아닌 항의 개수가 $N, M$ 일 때,  $O(N + M)$ 의 선형 시간이 소요됩니다.
* **특징**: 0인 항을 무시하므로 행렬의 실제 크기보다 데이터의 개수에 비례하는 빠른 연산이 가능합니다.

## 3-2. 희소 행렬의 단순 전치 (Naive Transpose)
행렬의 행($i$)과 열($j$)을 바꾸는 작업 자체는 매우 직관적입니다.

* **로직**: 각 항의 $(i, j, \text{value})$ 를 가져와서 $(j, i, \text{value})$ 로 순서만 바꿔 저장합니다.
* **수행 시간**: 항의 개수($N$)만큼 한 번만 순회하면 되므로 $O(N)$ 이 소요됩니다.
  - 예: $(0, 3, 22) \rightarrow (3, 0, 22)$

## 3-3. 브루트 포스(Brute-Force) 정렬 문제
단순 전치가 $O(N)$ 으로 끝난 후, **2-3에서 정한 '오름차순 저장 약속'이 깨지는 문제** 가 발생합니다.
이후의 정상적인 연산을 위해 **'오름차순 정렬'을 유지하는 비용**이 브루트 포스 방식에서는 $O(N^2)$ 까지 치솟게 됩니다.

### ① 저장 원칙을 지키기 위해 정렬한다면?

* **문제**: 전치된 결과물은 row의 순서가 뒤죽박죽입니다. 이를 다시 오름차순으로 맞추려면 기존 원소들을 이동시켜야 하는 상황이 발생합니다.
* **정렬 비용** $O(N^2)$ : 
  - 단순히 하나씩 뒤집은 후 다시 정렬 알고리즘을 돌리거나, 
  - 뒤집은 항을 넣을 때마다 적절한 위치를 찾아 다른 원소들을 밀어내며 삽입해야 합니다.
  - 이 과정에서 최악의 경우 $O(N^2)$ 의 시간이 소요됩니다.

### ② 저장 원칙을 위배하여 정렬하지 않고 연산에 사용한다면?
* 만약 전치된 행렬을 **정렬하지 않고 그대로** 덧셈 연산에 사용한다면? 
* "앞에서부터 훑으면 된다"는 약속이 깨졌기 때문에, 한 항을 더할 때마다 상대방 행렬의 모든 항을 뒤져야 합니다. 
* 결과적으로 효율적이던 덧셈마저 $O(N^2)$  급으로 느려지게 됩니다.

## 3-4. 행렬의 전치 - 열 우선 검색 방법 (Column-priority Search)
단순 전치 후 다시 정렬하는 번거로움을 피하기 위해, **처음부터 전치될 행렬의 행 순서(원래 행렬의 열 순서)대로 하나씩 찾아내어 복사**하는 방식입니다.
```java
public SparseMatrix transpose(int colSize, int rowSize, int termCnt) {
    if (termCnt > 0) { // 0이 아닌 원소가 있는 경우에만 실행
        int currentB = 0;
        for (int c = 0; c < colSize; c++) { // 원본 행렬의 각 열을 기준으로 순회
            for (int i = 0; i < termCnt; i++) { // 전체 0이 아닌 원소들을 확인
                // 현재 찾고 있는 열(c)과 일치하는 원소를 발견하면 전치 수행
                if (array[i].col == c) {
                    transMatrix.array[currentB].row = c;
                    transMatrix.array[currentB].col = array[i].row;
                    transMatrix.array[currentB++].value = array[i].value;
                }
            }
        }
    }
    return transMatrix;
}
```
### 알고리즘 로직
1. **외곽 루프**: 원래 행렬의 **열(Column)** 인덱스 0부터 `colSize - 1`까지 순차적으로 검사합니다.
2. **내부 루프**: 행렬의 모든 유효 항(`termCnt`)을 처음부터 끝까지 스캔합니다.
3. **조건 검사**: 현재 찾고 있는 열 번호와 일치하는 항이 있다면, 해당 항을 전치(`i, j` 교체)하여 결과 배열에 차례대로 담습니다.

### 특징:  
* **정렬 보장**: 작은 col 번호부터 순서대로 찾아내어 결과 배열에 넣기 때문에, 전치된 행렬은 자동으로 **행 우선 오름차순 약속**을 지키게 됩니다.
* **추가 공간 절약**: 별도의 복잡한 정렬을 위한 메모리 없이, 결과 행렬과 인덱스 변수만 있으면 되므로 공간 복잡도는 **$O(1)$** (고정된 추가 공간)입니다.

### 성능 분석 : 
* **총 실행 시간**: $O(\text{colSize} \times \text{termCnt})$
  - 열의 개수만큼 반복하면서, 매번 전체 항($N$)을 다 훑어야 합니다.
* **최악의 경우**: 항의 개수 termCnt가 `rowSize * colSize`인 밀집 행렬(Dense Matrix)이라면,  
  시간 복잡도는 $$O(\text{colSize} \times \text{termCnt}) \rightarrow O(\text{colSize} \times (\text{rowSize} \times \text{colSize}))$$  
   즉, **$O(\text{rowSize} \times \text{colSize}^2)$** 까지 치솟습니다.
* **평가**: "공간 절약을 위해 시간을 희생한 결과"입니다. 

## 3.5. Brute-Force vs Column-priority Search
Column-priority Search 방식이 일반적인 정렬 Brute-Force $O(N^2)$ 보다 효율적일 수 있습니다.

* **무거운 연산 없음**: Brute-Force 정렬은 데이터를 옮길 때마다 뒤의 데이터들을 밀어내야 하는 무거운 'Shifting'이 동반되지만, 이 방식은 단순 **비교($==$)와 복사**만 수행합니다.
* **$C \ll N$ 인 경우**: 만약 열의 개수가 데이터 개수보다 훨씬 적다면, $C \times N$은 $N^2$보다 훨씬 효율적인 수치가 됩니다.

### 4. 열 우선 검색(Column-priority Search)의 한계
이 방식에는 List 입장에서 보면 매우 **비효율적인 부분**이 존재합니다.

* **중복 스캔**: 0번 열을 찾으려고 리스트 $N$ 개를 다 훑었는데, 바로 다음 1번 열을 찾기 위해 **방금 본 리스트를 또 처음부터 끝까지** 뒤져야 합니다. 
    리스트에 있는 항들은 이미 본인들의 정보를 다 알고 있는데, 시스템이 이를 기억하지 못하고 열의 개수만큼 반복해서 물어보는 셈입니다.
* **밀집 행렬에서 최악 성능**: 행렬에 데이터가 꽉 차 있다면 복잡도가 $O(\text{row} \times \text{col}^2)$까지 치솟아, 차라리 일반 배열을 쓰는 것보다 훨씬 느려지는 비극이 발생합니다.

## 3-5. 행렬의 전치 - 빠른 전치 (Fast Transpose)

중복 스캔 문제를 해결하기 위해 **메모리를 조금 더 사용하여 시간을 획기적으로 단축**한 개선 알고리즘입니다.   
Fast Transpose는 데이터를 이동시키기 전에 **통계(개수 세기)**를 먼저 냄으로써,   
무작위로 흩어진 데이터를 단 한 번의 스캔만으로 정렬된 위치에 꽂아넣는 **인덱싱(Indexing)** 기법으로  
**각 항이 들어갈 위치를 미리 계산**하는 것입니다.

### 알고리즘 로직 (예약석 시스템)
원본 리스트를 여러 번 뒤지는 대신, 세 단계를 거쳐 한 번에 자리를 찾아줍니다.

1. **원소 수 구하기 (EACH_ROW_SIZE)**: 원본 행렬을 한 번 훑으며 각 열에 데이터가 몇 개씩 있는지 셉니다. (전치 행렬 입장에서는 각 행의 데이터 개수가 됩니다.)
2. **시작 위치 계산 (ROW_START)**: 위에서 구한 개수를 바탕으로, 전치 행렬의 각 행이 결과 배열의 **몇 번 인덱스부터 시작해야 하는지** 계산하여 저장합니다.
   - 예: 0행에 데이터가 2개 있다면, 1행의 시작 위치는 2번 인덱스가 됩니다.
3. **데이터 이동**: 원본 리스트를 **딱 한 번만 처음부터 끝까지 스캔**합니다. 각 항의 `col` 번호를 보고 미리 계산한 `ROW_START` 위치에 즉시 꽂아넣습니다. 항을 하나 넣을 때마다 해당 행의 `ROW_START` 값을 +1 해줌으로써 다음 항이 올 자리를 준비합니다.

### 시간 복잡도: $O(\text{colSize} + \text{termCnt})$
열의 개수($C$)만큼 시작 위치를 계산하고, 항의 개수($N$)만큼 데이터를 한 번씩만 옮깁니다.

* **비교**: 열 우선 검색의 $O(C \times N)$에서 $O(C + N)$ 으로 연산량이 기하급수적으로 줄어들었습니다.
* **성능**: 데이터가 아무리 많아져도 리스트를 중복해서 읽지 않으므로 매우 빠릅니다.

### Space-Time Trade-off
* **메모리 추가 사용**: `EACH_ROW_SIZE`와 `ROW_START` 정보를 담기 위한 추가적인 배열(공간)이 필요합니다.
* **가치**: 현대 컴퓨터 환경에서는 약간의 메모리를 더 써서(공간) 실행 속도를 비약적으로 높이는(시간) 이 방식이 훨씬 효율적입니다.

 ## 4-1. 희소 행렬 곱셈: "0은 계산할 가치도 없다"
희소 행렬 곱셈의 핵심은 **결과가 0이 될 것이 뻔한 연산은 아예 시작조차 하지 않는 것**입니다.

* **기본 원리**: $d_{ij} = \sum_{k=0}^{n-1} a_{ik}b_{kj}$ 에서 $a_{ik}$나 $b_{kj}$ 중 하나만 0이어도 결과는 0입니다. 
* **희소 행렬의 전략**: 데이터가 있는(Non-zero) 항들끼리 만나는 지점만 골라내어 계산합니다.

### 기존 곱셈(Dense) vs 희소 곱셈(Sparse)
가로세로 1,000칸($1000 \times 1000$)인 행렬에 데이터가 각각 딱 10개씩만 있다고 가정해 봅시다.

1. **기존 방식 (Dense)**:
    * **연산량**: $1000^3 =$ **10억 번**의 곱셈 수행.
    * **비효율**: 10억 번 중 의미 있는 계산은 단 100번 남짓이며, 나머지 **9억 9,999만 번은 $0 \times 0$** 는 당연히 결과가 0인 연산의 낭비.
2. **희소 행렬 방식 (Sparse)**:
    * **연산량**: 오직 데이터가 존재하는 항들끼리 매칭될 때만 곱셈을 수행합니다.
    * **효율**: 0인 칸은 메모리에서 읽지도 않으므로, 연산량이 행렬의 전체 크기( $N$ )가 아닌 **데이터의 개수($terms$)에 비례**하여 급감합니다.

## 4-2. Fast Transpose 의 행렬 곱셈에서의 활용

* **데이터 구조의 한계 (Row-major)**
   - 우리 희소 행렬은 Triple 방식, **행(row) 순서**로 정렬되어 저장되어 있습니다.
   - 따라서 특정 '행'의 원소들을 읽는 것은 매우 쉽고 빠릅니다 ($O(1)$ 수준).
   - 하지만 특정 **'열(column)'**의 원소들을 찾으려면? 리스트의 처음부터 끝까지 다 뒤져야 합니다.

* **곱셈의 규칙: (A의 행) × (B의 열)**: 
  행렬 곱셈 공식($A$의 행 $\times$ $B$의 열)을 수행하려면 **B의 '열' 데이터**가 계속 필요합니다.
  B가 $1,000$개의 열을 가졌다면, 전치를 안 할 경우 B의 리스트 전체를 **1,000번이나 반복해서 스캔**해야합니다.
  B의 열을 하나 찾을 때마다 리스트를 처음부터 끝까지 스캔하는 끔찍한 비효율이 발생합니다.

* **전치를 통한 해결 (B → $B^T$)**: 
  연산 시작 전, **Fast Transpose**를 딱 한 번 실행하여 B를 전치해버리면, 원본 B의 열들이 $B^T$의 행으로 변합니다.
  이제 $B^T$도 '행 순서'로 정렬되어 있으니, 리스트를 순차적으로 읽기만 해도 B의 열 데이터를 쏙쏙 뽑아낼 수 있습니다.

* **최적화 효과**: 
   **데이터 성질 변화**: 원본 B의 **열(Column)** 들이 $B^T$에서는 **행(Row)** 이 됩니다.
   **순차 접근(Linear Scan)**: 이제 A도 '행' 순서, $B^T$도 '행' 순서로 정렬되어 있습니다. 

## 4-3. 알고리즘 수행 단계 (Row-by-Row 합병)
1. **전치 단계**: 행렬 B를 Fast Transpose하여 $B^T$를 생성합니다.
2. **행 선택**: 행렬 A의 $i$번째 행에 속한 항들을 가져옵니다.
3. **매칭 및 곱셈**:
    * $B^T$의 각 행 $j$(원본 B의 $j$열)와 A의 행 $i$를 비교합니다.
    * A의 항의 열 번호($k$)와 $B^T$의 항의 열 번호(원본 B의 행 번호 $k$)가 일치하는 경우에만 곱셈을 수행합니다.
4. **결과 누적**: 곱한 값들을 임시 버퍼에 더해두었다가, 한 행의 계산이 모두 끝나면 0이 아닌 값들만 결과 행렬에 순서대로 저장합니다.

## 4-4. 성능 분석:

* **통상적인 곱셈 복잡도**: $O(rows_A \times cols_A \times cols_B) \approx O(N^3)$
* **희소 행렬 곱셈 복잡도**: $O(cols_B \times terms_A + rows_A \times terms_B)$

> **결론**: 데이터가 희소할수록(0이 많을수록) $terms$ 값이 행렬 전체 크기($N^2$)보다 훨씬 작아집니다.  
>  따라서 희소 행렬 알고리즘은 거대한 행렬 연산에서도 현실적으로 처리가능한 수준의 압도적인 속도를 보여줍니다.
