## Array 응용 문제 - 다항식(Polynomial) 표현

---
## 1-1. 다항식 표현 방법 1 - 배열 인덱스 매핑

다항식을 컴퓨터 메모리에 저장하는 가장 직관적인 방법은  
**배열의 물리적 위치(인덱스)를 수학적 의미(지수)로 직접 매핑**하는 것입니다.

이 방식은 "배열의 인덱스(Index) = 항의 지수(Exp)"라는 원칙을 따릅니다.

* **구조**: `coef[i]`에는 $x^i$ 항의 계수(Coef)를 저장합니다.
* **배열 크기**: 최고 차수(`maxDegree`)를 기준으로 `maxDegree + 1`개의 공간을 할당합니다 (0차항부터 시작하기 때문).

### 예시
다항식 $b(x) = x^8 - 10x^5 - 3x^3 + 1$ 의 저장 형태:
* `coef[8] = 1`, `coef[5] = -10`, `coef[3] = -3`, `coef[0] = 1`
* 값이 없는 $x^7, x^6, x^4, x^2, x^1$ 항의 인덱스에는 모두 **0**이 저장됩니다.
* **핵심:** $x^7, x^6$ 처럼 존재하지 않는 항도 **`coef[7] = 0`, `coef[6] = 0`과 같이 배열 칸을 차지하며 저장**됩니다.

## 1-2. 다항식의 덧셈 과정 (Add)

배열 기반 다항식 덧셈은 다음과 같은 단계로 이루어집니다.

1.  **결과 차수 결정**: 더하려는 두 다항식 중 더 큰 차수를 결과 다항식의 기본 차수로 정합니다.
2.  **순차적 합산 (Looping)**: 인덱스 `0`부터 최고 차수까지 루프를 돌며 각 위치의 계수를 더합니다.
    * `Result.coef[i] = A.coef[i] + B.coef[i]`
3.  **범위 체크**: 만약 한쪽 다항식의 차수가 낮아 인덱스 범위를 벗어나면, 해당 항의 값은 **0**으로 간주하여 계산을 지속합니다.
4.  **최종 최적화**: 덧셈 결과 maxDegree의 coef가 `0`이 된 경우  
   $3x^2 + (-3x^2)$ 와 같은 경우, 실제 유효한 maxDegree를 다시 찾아 데이터를 정돈합니다.

## 1-3. 장점과 단점

### 장점: 밀집 다항식(Dense Polynomial)에 최적화
* **알고리즘의 단순성**: 지수를 별도로 비교할 필요 없이 인덱스 기반으로 반복문만 돌면 되므로 덧셈, 뺄셈, 곱셈 등의 구현이 매우 간단합니다.
* **빠른 접근 (Random Access)**: 특정 지수($x^i$)의 계수를 확인하거나 수정할 때 인덱스를 사용하여 **$O(1)$** 시간에 즉시 접근할 수 있습니다.
* **밀집 다항식에서의 효율성**: 거의 모든 항에 계수가 존재하는 **밀집 다항식**의 경우, 메모리 빈틈 없이 데이터를 꽉 채워 관리할 수 있어 가장 효율적인 방식입니다.

### 단점: 희소 다항식(Sparse Polynomial)에서의 한계
* **공간 낭비**: 항은 몇 개 없는데 차수만 높은 **희소 다항식**의 경우 메모리 효율이 극도로 떨어집니다.
    * 예: $x^{1000} + 1$을 표현하려면 1001개의 배열 칸이 필요하지만, 실제 값은 2개뿐이고 나머지 999개는 의미 없는 `0`으로 채워져 공간을 차지합니다.
* **연산 성능 저하**: 실제 데이터가 없는 항(계수가 0인 항)에 대해서도 항상 최고 차수만큼 반복문을 돌아야 합니다.
    * 희소 다항식끼리 더할 때도 실제 항의 개수와 상관없이 시간 복잡도는 항상 $O(\text{maxDegree})$가 되어 불필요한 연산이 발생합니다.

## 2-1. 다항식 표현 방법 2 : 계수가 0이 아닌 (계수, 지수) 쌍의 나열

표현 1(배열 인덱스 매핑)의 치명적인 단점인 **메모리 낭비**를 해결하기 위해, 값이 0이 아닌 항들만 따로 모아 관리하는 방식입니다.
이 방식은 인덱스에 지수 정보를 숨기지 않고, **지수(Exp)를 데이터 필드에 직접 명시**합니다.

* **구조**: `Term`이라는 별도의 클래스를 정의하여 (계수, 지수) 쌍을 저장합니다.
* **배열의 역할**: `Term` 객체들을 담는 그릇일 뿐, **배열의 인덱스는 지수와 아무런 상관이 없습니다.**
* **저장 원칙**: 계수가 `0`이 아닌 항들만 **지수 내림차순**으로 빈틈없이 정렬하여 저장합니다.

### 예시
다항식 $b(x) = x^8 - 10x^5 + 1$ 의 저장 형태:
* **표현 1**: 크기 9의 배열 필요 (`coef[7], coef[6]` 등 0인 항도 공간 차지)
* **표현 2**: 단 3개의 `Term` 객체만 필요
    * `termArray[0] = (1, 8)`
    * `termArray[1] = (-10, 5)`
    * `termArray[2] = (1, 0)`


## 2-2. 다항식의 덧셈 과정 (Add) : 두 포인터 (Two Pointers) 합치기

지수가 인덱스와 일치하지 않기 때문에, 두 다항식을 더할 때는 '병합 정렬(Merge Sort)'과 유사한 알고리즘을 사용합니다.

1.  **두 개의 포인터**: 각 다항식의 현재 검사 위치를 가리키는 포인터(`idx`)를 둡니다.
2.  **지수 비교**: 두 포인터가 가리키는 항의 지수를 비교합니다.
    * **지수가 큰 쪽**: 결과 다항식에 먼저 넣고 해당 포인터를 한 칸 전진합니다.
    * **지수가 같은 경우**: 두 계수를 합산합니다. 합이 `0`이 아니면 결과에 넣고, 두 포인터 모두 전진합니다.
3.  **나머지 처리**: 한쪽 다항식이 먼저 끝나면, 남은 다항식의 항들을 순서대로 모두 결과에 담습니다.

## 2-3. 장점과 단점

### 장점: 희소 다항식(Sparse Polynomial)에서의 압도적 효율
* **메모리 절약**: $x^{1000} + 1$ 처럼 차수는 높지만 항이 적은 경우, 딱 2개의 공간만 사용하므로 메모리 낭비가 거의 없습니다.
* **연산 시간 단축**: 0인 항을 훑을 필요가 없습니다. 덧셈 연산 시 최고 차수가 아무리 높아도 실제 항의 개수(N, M) 만 비례하는 시간 복잡도 $O(N + M)$ 을 가집니다.
   * 배열 확장(Resizing)의 영향: 중간에 배열 크기를 2배로 늘리는 작업이 발생하더라도,  
     이는 전체 실행 시간에 비해 아주 작은 상수 수준의 영향만 미치므로 전체적인 효율성을 해치지 않습니다. Amortized $O(1)$ .

### 단점 :
* **관리의 복잡성**: 별도의 `Term` 클래스가 필요하며, 배열의 크기를 동적으로 늘려주는 `Resizing` 로직이나 지수 비교 로직이 추가되어 구현이 더 까다롭습니다.
* **밀집 다항식에서의 오버헤드**: 모든 항이 꽉 차 있는 밀집 다항식의 경우, 계수뿐만 아니라 지수 정보까지 저장해야 하므로 오히려 표현 1보다 메모리를 더 많이 쓸 수도 있습니다.

## 2-4. 정렬의 중요성과 "개발자 간의 약속"
표현 2가 고성능 O(m+n) 을 낼 수 있는 근본적인 이유는 데이터가 특정 질서(정렬)에 의해 관리되기 때문입니다.

### ① 지수 내림차순 정렬의 힘
* Two Pointers 방식은" 두 다항식이 이미 지수 순서대로 정렬되어 있다" 는 전제 하에 작동합니다.
* **만약 정렬되어 있지 않다면?** 항을 하나 더할 때마다 상대 배열 전체를 뒤져야 하므로 시간 복잡도는 $O(m*n)$으로 급격히 악화됩니다.

### ② 개발자 간의 "약속 지키기"
* **클래스 설계의 철학**: 효율적인 클래스를 만들려면 "모든 항은 항상 지수 순서대로 저장한다"는 내부 규칙을 설계 단계에서 확정해야 합니다.
* **약속의 중요성**: 이 자료구조를 사용하는 모든 개발자가 이 규칙(약속)을 지킬 때만 고성능 알고리즘이 성립합니다. 
* 결국, **좋은 설계란 단순히 코드를 잘 짜는 것이 아니라, 효율성을 보장하는 "규칙"을 정의하고 이를 엄격히 준수하는 것**입니다.

## 3. 표현 1 vs 표현 2 비교

| 구분 | 표현 1 (단순 배열) | 표현 2 (Term 객체 배열) |
| :--- | :--- | :--- |
| **저장 방식** | 인덱스 = 지수 | (계수, 지수) 객체 저장 |
| **핵심 지표** | 최고 차수 (maxDegree) | 실제 항의 개수 (Term Count) |
| **공간 복잡도** | $O(\text{maxDegree})$ | $O(\text{Term Count})$ |
| **시간 복잡도(덧셈)** | $O(\max(\text{degree}_a, \text{degree}_b))$ | **$O(m + n)$** |
| **최적화 대상** | 모든 항이 꽉 찬 **밀집 다항식** | 항이 듬성듬성한 **희소 다항식** |
| **필수 조건** | 없음 | **지수 내림차순 정렬 (약속)** |
